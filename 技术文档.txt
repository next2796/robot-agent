第1章 项目概述

1.1 项目背景

随着人工智能技术的快速发展，智能对话系统在医疗领域的应用越来越广泛。本项目旨在构建一个基于大语言模型的医疗咨询对话机器人，通过自然语言处理技术为用户提供初步的医疗症状咨询、疾病信息查询和风险评估服务。该系统利用 Ollama 本地部署的大语言模型 gemma3:1b 作为核心智能引擎，结合结构化的医疗知识库，实现智能化的医疗咨询交互。

1.2 项目目标

本项目的主要目标是构建一个轻量级、可部署的医疗咨询对话系统，该系统能够通过 Web 界面与用户进行自然语言交互，理解用户的咨询意图，从医疗知识库中检索相关信息，并结合大语言模型生成通俗易懂的回复。同时，系统会根据症状的严重程度进行风险评估，对高风险情况给出明确的就医提示。

1.3 技术选型

本项目的技术选型充分考虑了轻量化、易部署和本地化运行的需求。核心开发语言为 Python 3.14，后端框架采用 Flask 3.1.3，前端使用原生 HTML/CSS/JavaScript 构建单页面应用，数据库采用 SQLite 3，轻量级且无需额外配置。大语言模型方面选用 Ollama 部署的 gemma3:1b，该模型体积小、性能适中，适合本地部署场景。


第2章 系统架构

2.1 整体架构设计

系统采用经典的三层分布式架构设计，包括表现层、业务逻辑层和数据访问层。表现层负责用户界面交互，业务逻辑层处理核心业务规则，数据访问层管理数据持久化。这种分层架构使得系统各部分职责清晰，便于维护和扩展。前端采用 HTML5 单页面应用，通过 AJAX技术与后端 API 网关进行通信。后端使用 Flask 框架构建 RESTful API 服务，实现对话管理、意图识别、知识库检索等功能。数据库集群采用 SQLite，内置疾病知识库表，存储疾病名称、症状、建议和风险等级信息。

2.2 技术栈说明

（a）后端技术栈

Python 3.14：作为核心开发语言，提供简洁高效的语法和丰富的生态系统。Flask 3.1.3：轻量级 Web 框架，灵活可扩展，适合构建 RESTful API 服务。Flask-CORS 6.0.2：跨域资源共享扩展，解决前端调用时的跨域问题。SQLite 3：嵌入式关系型数据库，无需独立部署，适合中小型应用场景。Requests：Python HTTP 客户端库，用于与 Ollama API 进行通信。

（b）前端技术栈

HTML5：超文本标记语言，构建页面结构和内容。CSS3：层叠样式表，实现界面美化、布局和交互效果。JavaScript（原生）：实现前端逻辑控制、异步请求和 DOM 操作。

（c）大模型技术栈

Ollama：本地大语言模型运行框架，支持多种开源模型的本地部署。gemma3:1b：Google 开发的轻量级大语言模型，参数量小但性能良好，适合本地部署。


第3章 对话流程设计

3.1 整体对话流程

本系统采用流水线式的对话处理流程，将用户输入依次经过意图识别、槽位填充、知识库检索、风险评估和回复生成五个处理阶段。每个阶段的结果会影响后续阶段的处理逻辑，最终生成综合性的回复内容。这种流程设计既保证了处理的一致性，又便于各环节的独立优化和调试。

3.2 医疗咨询场景对话流程详解

以用户输入"我发烧了，还咳嗽"为例，详细说明完整对话流程的处理过程。

（1）用户发起咨询请求

用户在 Web 前端的输入框中输入"我发烧了，还咳嗽"，点击发送按钮或按回车键。前端 JavaScript 监听用户输入事件，获取输入内容并发送到后端 API。

（2）意图识别阶段

后端接收到用户输入后，构造意图识别 Prompt 并调用 Ollama 模型。模型根据输入内容判断用户意图，本例中识别结果为 symptom_consult，表示用户正在咨询症状。

（3）槽位提取阶段

根据识别出的意图类型，调用相应的槽位提取逻辑。对于症状咨询意图，模型从输入中提取症状关键词"发烧"和"咳嗽"，形成槽位字典 {"symptoms": "发烧, 咳嗽"}。

（4）知识库匹配阶段

使用提取出的症状关键词查询医疗知识库。系统执行 SQL 语句，在 diseases 表中查找 symptoms 字段包含"发烧"和"咳嗽"的记录。本例中匹配到"感冒"记录，返回完整的疾病信息。

（5）风险评估阶段

根据知识库返回的疾病记录中的 risk_level 字段值，生成相应的风险提示。本例中感冒的风险等级为"低"，因此生成低风险提示信息。

（6）回复生成阶段

将用户输入、知识库信息和风险提示整合，构造完整的 Prompt 发送给 Ollama 模型。模型综合这些信息，生成通俗易懂的自然语言回复，包含疾病解释、护理建议和风险提示。

（7）返回结果

后端将处理结果封装为 JSON 格式返回给前端，包括意图类型、槽位信息、知识库记录、风险提示和最终回复内容。


第4章 第三方服务调用

4.1 Ollama 本地模型服务

4.1.1 服务概述

Ollama 是一个开源的本地大语言模型运行框架，它允许用户在本地环境中部署和运行各种开源大语言模型，无需依赖云端 API 服务。Ollama 提供了简洁的命令行接口和 API 接口，支持模型的下载、运行和管理。本项目使用 Ollama 部署 gemma3:1b 模型，作为系统的核心智能引擎，处理意图识别、槽位填充和回复生成等自然语言处理任务。

4.1.2 服务部署

Ollama 服务的部署过程包括安装 Ollama 客户端、下载模型和启动服务三个步骤。首先，用户需要从 Ollama 官方网站下载并安装 Ollama 客户端程序。然后，通过命令行执行 ollama pull gemma3:1b 下载模型文件。最后，执行 ollama run gemma3:1b 或 ollama serve 启动模型服务。Ollama 默认监听本地 11434 端口，提供 RESTful API 接口供外部调用。

4.1.3 API 调用方式

Ollama 提供了简洁的 HTTP API 接口，后端通过 Python requests 库调用该接口。以下是 call_ollama 函数的核心实现代码：

    def call_ollama(prompt):
        """调用本地Ollama模型"""
        try:
            response = requests.post(
                OLLAMA_API,
                json={
                    "model": MODEL_NAME,
                    "prompt": prompt,
                    "stream": False
                },
                timeout=10
            )
            return response.json().get("response", "")
        except Exception as e:
            print(f"模型调用失败: {e}")
            return None

    OLLAMA_API = "http://localhost:11434/api/generate"
    MODEL_NAME = "gemma3:1b"

4.1.4 并入项目架构

Ollama 服务在逻辑上作为独立的自然语言处理引擎嵌入到系统的业务逻辑层。系统架构中，NLP 引擎模块通过 HTTP 协议与 Ollama 服务通信，发送 Prompt 并接收模型生成的回答。这种设计实现了核心智能逻辑与业务系统的解耦，当需要升级或更换模型时，只需修改 OLLAMA_API 和 MODEL_NAME 配置，无需改动业务代码。

4.1.5 服务调用场景

本项目在三个场景中调用 Ollama 模型服务：意图识别场景中，构造包含用户输入和意图类型选项的 Prompt，模型返回分类结果；槽位填充场景中，根据不同意图类型构造针对性的提取 Prompt，模型返回提取的实体信息；回复生成场景中，将用户输入、知识库信息和风险提示整合为完整 Prompt，模型生成最终的自然语言回复。

第6章 总结与展望

6.1 项目总结

本项目成功实现了一个基于大语言模型的医疗咨询对话机器人系统。系统采用 Flask 框架构建后端服务，使用 SQLite 数据库存储医疗知识库，通过 Ollama 部署 gemma3:1b 模型实现自然语言处理能力。系统实现了症状咨询、疾病查询和风险评估三大核心功能，能够为用户提供初步的医疗参考信息。测试结果表明，系统功能完整、运行稳定、响应及时，具备一定的实用价值。

6.2 改进方向

（a）知识库扩展

当前知识库仅包含三条示例疾病记录，数据规模有限。可以考虑引入更完整的医学知识图谱，涵盖更多疾病类型、症状表现和治疗方案，提高系统的咨询能力。

（b）模型升级

gemma3:1b 作为轻量级模型，在复杂语境理解方面存在局限。可以考虑升级到更大规模的模型如 gemma3:4b 或其他开源模型，提升对话质量和准确性。

（c）多轮对话支持

当前系统仅支持单轮对话交互，无法记忆对话历史。可以引入对话状态管理机制，支持多轮连续对话，提供更连贯的用户体验。

（d）用户反馈机制

增加用户满意度反馈功能，收集用户对回复的评价，用于持续优化系统性能和回复质量。
